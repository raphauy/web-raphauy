# /discover — Generate project data.json from a repository

## Usage
```
/discover /path/to/repo --slug my-project
/discover                                   # auto-picks next from github-repos.md
```

The raw arguments are: `$ARGUMENTS`

### Argument resolution

**If arguments are provided**, parse them as: `/path/to/repo --slug slug-name`
- First part: path to the repository to analyze
- `--slug`: the slug to use for the project (required)

**If no arguments are provided** (empty or blank), auto-detect the next project:
1. Read `docs/github-repos.md` (in the portfolio repo)
2. Look only at the **"Repos locales"** table (all repos there are cloned at `~/desarrollo/<repo-name>`)
3. Find the first row where the **Portfolio** column is empty (not `x`)
4. Use the **Repo** column as the slug and `~/desarrollo/<repo-name>` as the path
5. Announce which project was auto-selected before proceeding

## Instructions

You are generating a `data.json` file for a portfolio project. Follow these steps carefully:

### Step 1 — Gather context from the repository

Read the following files from the target repo (skip any that don't exist):
1. **package.json** — read `name`, `dependencies`, `devDependencies` to infer tech stack
2. **README.md** — understand project purpose and features
3. **PRD.md** — detailed product requirements if available
4. **CLAUDE.md** — project context and conventions

Run **`cloc --json .`** in the target repo directory to count lines by language. If `cloc` is not available, try `tokei -o json` or `scc --format json` as alternatives.

Detect AI usage:
- Check for files: `CLAUDE.md`, `.cursor/`, `.github/copilot`, `.aider*`
- Search git log: `git log --oneline --all -500 | grep -iE "Co-Authored-By: Claude|Generated by|AI|copilot|cursor|aider"` (first 500 commits)

Infer `startDate` from first commit: `git log --reverse --format=%cs | head -1` (use as-is, YYYY-MM-DD format)

6. **GitHub stats** — Always run this step:
   - Get the GitHub URL from the repo's remote: `git -C <repo-path> remote get-url origin`
   - Run `pnpm github-data <github-url>` **from the portfolio repo directory** (`/home/raphael/desarrollo/web-raphauy.dev/`) to fetch commit history and repo stats
   - This populates `data/github-activity.json` with commit data for the activity grid
   - Also use this URL as the `repoUrl` field in the generated `data.json`

### Step 2 — Generate structured data

Using all the gathered context, generate the following fields:

```json
{
  "slug": "<the provided slug>",
  "name": "<human-readable project name, not the npm name>",
  "tagline": "<one sentence, max 80 chars, describes what it does>",
  "description": "<2-4 sentences, oriented towards a potential client>",
  "highlights": ["<3-5 key features/achievements>"],
  "type": "<web-app|mobile-app|api|cli|library|landing|other>",
  "tags": ["<3-8 relevant tags, lowercase, don't duplicate tech names>"],
  "tech": ["<main technologies from dependencies + code>"],
  "startDate": "<YYYY-MM-DD from first commit>",
  "endDate": null,
  "stats": {
    "languages": {"<Language>": <lines>},
    "totalLines": <total>
  },
  "url": null,
  "repoUrl": "<GitHub URL from git remote>",
  "screenshots": [],
  "builtWithAI": <true|false>,
  "aiDetails": "<brief description of AI usage or null>"
}
```

Rules:
- For `stats.languages`, only include languages with >50 lines
- For `tags`, use lowercase, don't repeat tech names, focus on domain/category
- `type` should reflect the primary nature of the project
- Write `description` and `tagline` in **English**

#### `tech` field — IMPORTANT

The `tech` array is NOT just npm package names. It represents the **technologies, services, and integrations** that are impressive or relevant to show in a portfolio. Go beyond `package.json`:

1. **Frameworks & libraries from dependencies**: Next.js, React, Tailwind CSS, Prisma, tRPC, etc.
2. **UI component libraries**: If `shadcn` or `@radix-ui` or `@base-ui` is in dependencies, include `shadcn/ui`. If `@mui` include `Material UI`. Always check for these.
3. **External services & APIs**: Look at the code, README, and env variables for integrations like Stripe, OpenAI, Clerk, Auth.js, Resend, Twilio, etc.
4. **Domain-specific integrations**: If the project integrates with WhatsApp, Telegram, Slack, etc., include them (e.g. "WhatsApp", "Telegram Bot"). Look for keywords in code, README, and env files.
5. **Databases & infrastructure**: PostgreSQL, Redis, MongoDB, Vercel, AWS, Docker, etc. Check `docker-compose.yml`, env vars for DB URLs, and dependencies.
6. **AI/ML tools**: OpenAI, Anthropic, LangChain, Vercel AI SDK, etc.

Scan these sources to detect tech:
- `package.json` dependencies and devDependencies
- `README.md`, `PRD.md`, `CLAUDE.md` for mentioned technologies
- `.env.example` or `.env.local.example` for service API keys (e.g. `OPENAI_API_KEY`, `STRIPE_SECRET_KEY`, `WHATSAPP_TOKEN`)
- `docker-compose.yml` for infrastructure services
- Import statements in source code (sample 5-10 key files like `src/lib/`, `src/app/api/`, `src/services/`)

Order the array by relevance: primary framework first, then key libraries, then services/integrations.

### Step 3 — Write/update data.json

The target file is: `projects/<slug>/data.json` (relative to the portfolio repo at `/home/raphael/desarrollo/web-raphauy.dev/`)

**If the file already exists:**
- Read the existing file
- **Preserve** these manual fields from the existing file: `screenshots`, `url`, `repoUrl`, `startDate`, `endDate`
- **Update** all other fields with freshly generated data
- Merge intelligently

**If the file doesn't exist:**
- Create `projects/<slug>/data.json` with all fields
- Create `projects/<slug>/screenshots/` directory

### Step 4 — Update github-repos.md

After writing `data.json`, update `docs/github-repos.md`:
- Find the row matching the repo name (slug) in the table
- Set the **Portfolio** column to `x`

### Step 5 — Report

After writing the file, show the user:
1. A summary of what was detected
2. The generated `data.json` content
3. Which fields need manual review (especially: `url`, `repoUrl`, `screenshots`, `startDate`, `endDate`)
4. Remind them to add screenshots to `public/projects/<slug>/screenshots/`
5. If GitHub data was fetched, show the stats (stars, forks, total commits) and note that commit data was saved to `data/github-activity.json`
